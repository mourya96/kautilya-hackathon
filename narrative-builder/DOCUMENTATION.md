# Narrative Builder ŌĆö Code & Workflow Documentation

This document explains every module in the project, what it does, and how the pieces fit
together into the end-to-end workflow: **load ŌåÆ embed/index ŌåÆ retrieve ŌåÆ generate (RAG) ŌåÆ
validate ŌåÆ assemble ŌåÆ (optionally) evaluate**.

---

## 1. High-level architecture

The system turns a large news dataset + a user topic into a structured **narrative** (summary,
timeline, thematic clusters, and a relationship graph). The summary and timeline reasoning are
generated by a **local LLM served with vLLM**, grounded strictly in retrieved articles, and
orchestrated as a **LangGraph** corrective-RAG pipeline. **RAGAS** measures how faithful the
output is to the retrieved evidence (i.e. how little it hallucinates).

## 2. Module-by-module reference

### `config.py` ŌĆö configuration manager
- **`Config`** loads settings from defaults ŌåÆ `.env` file ŌåÆ `NARRATIVE_*` OS env vars (later
  sources win), validates them, and exposes `get(key, default)` / `set(key, value)`.
- `DEFAULTS` is the single source of truth for every tunable: data loading, embeddings, FAISS,
  search, clustering, narrative, **LLM/vLLM**, **RAGAS**, performance, output.
- `get_config()` returns a cached singleton; `reload_config()` rebuilds it (used by `--env`).
- `print_config()` / `save_to_file()` group keys into labelled categories for `--config`.
- **Key groups added for RAG/eval:** `USE_LLM`, `VLLM_BASE_URL/API_KEY/MODEL`,
  `LLM_TEMPERATURE/MAX_TOKENS/STREAMING/TIMEOUT`, `MAX_GENERATION_ATTEMPTS`, `RAG_CONTEXT_CHARS`,
  `RAGAS_JUDGE_*`, `EVAL_SET_PATH/TOP_K`, `EVAL_RESULTS_PATH`.

### `data_loader.py` ŌĆö dataset ingestion
- **`DataLoader.load_and_filter(min_rating)`** streams the JSON with `ijson` (handles arrays and
  `{items|articles|data|news: [...]}`), filtering by source rating; falls back to a standard
  `json.load` if streaming yields nothing.
- **`_clean_article`** normalises each record into the canonical dict the rest of the app relies
  on: `id, headline, content, full_text, date, url, source, source_rating, author, category,
  tags, raw`. `full_text = "headline. content"` is what gets embedded and shown to the LLM.
- `_parse_date` normalises many date formats to ISO; `get_articles()` returns the cleaned list.

### `embedder.py` ŌĆö semantic retrieval ("R" in RAG)
- **`NewsEmbedder`** wraps a `sentence-transformers` model + a **FAISS** index.
- `build_index(articles)` embeds `full_text` (batched, normalised), **caches embeddings** to
  `cache/embeddings.pkl`, and builds a FAISS index (`flat` for <100k docs, `ivf` for larger;
  optional GPU).
- **`search(query, top_k)`** embeds the query and returns the top-k articles, each annotated with
  a `similarity_score`. This is the retriever the RAG graph calls.
- Also provides `find_similar_articles`, `compute_similarity_matrix`, `batch_encode`.

### `llm_client.py` ŌĆö vLLM connection + streaming
- **`get_chat_llm(role)`** builds a `langchain_openai.ChatOpenAI` pointed at the vLLM
  OpenAI-compatible endpoint. `role="generator"` uses `VLLM_*`; `role="judge"` uses
  `RAGAS_JUDGE_*` (falling back to the generator settings). The client is **not** put in global
  streaming mode ŌĆö streaming is scoped to a single call (below).
- **`_resolve_api_key`** resolves the auth token with precedence: `.env` value ŌåÆ
  `VLLM_API_KEY` env var ŌåÆ `OPENAI_API_KEY` env var ŌåÆ `"EMPTY"` (for local, unauthenticated vLLM).
- **`stream_to_text(llm, messages, on_token)`** streams exactly one completion via `llm.stream()`,
  invoking `on_token` per chunk and returning the full text. This is the *only* place streaming
  happens, so the graph and the judge are unaffected.
- **`extract_json(text)`** robustly pulls the first JSON object out of a model reply (handles
  ```json fences and prose), so generation is resilient to models that ignore `response_format`.

### `rag_graph.py` ŌĆö the LangGraph corrective-RAG pipeline ("G" in RAG)
The heart of the new system. Builds a `StateGraph` whose state (`RAGState`) carries
`topic, top_k, articles, summary, timeline, attempts, grounded, feedback`.

**Nodes & edges**
1. **`retrieve`** ŌĆö calls the injected retriever (`NewsEmbedder.search`) ŌåÆ `articles`, resets
   `attempts`.
2. **`generate`** ŌĆö builds a **numbered context block** (`[1] (id=ŌĆ”, date=ŌĆ”, source=ŌĆ”) textŌĆ”`),
   sends the anti-hallucination system prompt + instructions, and asks for a single JSON object
   `{"summary", "timeline":[{"id","why_it_matters"}]}`. Streams tokens (live stderr progress)
   when `LLM_STREAMING` is on, else uses `invoke`. On a retry it appends the previous attempt's
   `feedback`.
3. **`validate`** ŌĆö runs the pure **`check_grounding`** guard: every `[n]` citation must be in
   range and every timeline `id` must be a retrieved article id. Sets `grounded` + `feedback`.
4. **conditional edge** ŌĆö if not grounded and `attempts < MAX_GENERATION_ATTEMPTS`, loop back to
   `generate`; otherwise drop anything still invalid and go to `END`.

**Helpers / API**
- `format_contexts(articles, max_chars)` ŌĆö the per-article context strings (reused by evaluation).
- `check_grounding(articles, summary, timeline)` ŌĆö standalone, LLM-free guard (unit-tested).
- **`RAGNarrator(retriever, llm)`** ŌĆö compiles the graph; `generate(topic, top_k)` runs it via
  `app.invoke` and returns `{topic, summary, timeline, articles}`.

> **Why a graph?** The retrieveŌåÆgenerateŌåÆvalidate loop with a conditional retry is exactly the
> "corrective RAG" pattern. Modelling it as a graph makes the self-correction explicit and easy to
> extend (e.g. add a re-retrieval node) ŌĆö and it directly serves the hallucination-reduction goal.

### `narrative_engine.py` ŌĆö narrative assembly
- **`NarrativeEngine(articles, topic, llm_narrative=None)`** produces the four output sections.
- `generate_summary()` ŌĆö returns the **grounded LLM summary** when `llm_narrative` is present,
  else a heuristic statistical summary.
- `build_timeline()` ŌĆö chronological entries; each `why_it_matters` uses the **LLM per-article
  reasoning** (matched by article `id`) when available, else the heuristic `_generate_importance_reason`.
- `create_clusters()` and `build_narrative_graph()` ŌĆö **unchanged heuristics** (keyword clustering;
  relationship edges like `builds_on`, `contradicts`, `adds_context`, `escalates`). These never
  depend on the LLM.

### `narrative_builder.py` ŌĆö orchestrator & CLI
- **`NarrativeBuilder(data_path, min_rating, use_llm)`** ties everything together:
  `load_and_filter()` ŌåÆ `initialize_embedder()` ŌåÆ `generate_narrative(topic, top_k)`.
- `_get_narrator()` lazily builds a `RAGNarrator` backed by `embedder.search`.
- `generate_narrative()`:
  1. If `use_llm`: run the RAG graph; take its `articles` + grounded `summary/timeline`.
  2. **Graceful fallback** ŌĆö if vLLM is unreachable or returns an empty summary, log a warning and
     fall back to plain retrieval + heuristic narrative (no crash).
  3. Build `NarrativeEngine` and emit the final dict with a `generation_mode` field
     (`rag_vllm` or `heuristic`).
- **CLI flags:** `--topic`, `--data`, `--rating`, `--top-k`, `--output`, `--env`, `--config`,
  `--save-config`, and the new `--no-llm`, `--evaluate`, `--eval-set`. `--evaluate` builds the
  index then hands off to `evaluation.evaluate_narratives` and exits.

### `evaluation.py` ŌĆö RAGAS hallucination evaluation
- **`load_eval_set(path)`** reads the JSONL golden set (`{"topic","reference"}` per line).
- **`_build_samples`** runs the RAG narrator for each topic and assembles RAGAS rows:
  `user_input=topic`, `response=summary`, `retrieved_contexts=format_contexts(articles)`,
  `reference=golden answer`.
- **`evaluate_narratives(eval_set_path, builder)`** builds a `ragas.EvaluationDataset` and runs
  **faithfulness, answer relevancy, context precision, context recall**. The judge LLM is
  `get_chat_llm(role="judge")` wrapped in `LangchainLLMWrapper`; embeddings reuse `EMBEDDING_MODEL`
  via `HuggingFaceEmbeddings` + `LangchainEmbeddingsWrapper` (all local, no external API). Writes
  per-query + aggregate scores to `eval_results.json` and prints a summary table.
- Metric intuition for hallucination: **faithfulness** = fraction of answer claims supported by
  the retrieved context; **answer relevancy** = answer addresses the topic; **context
  precision/recall** = retrieval quality vs. the reference.

### `utils.py` ŌĆö helpers
`setup_logging` (stderr logger, quiets noisy libs), `format_json_output` (pretty JSON),
`calculate_statistics`, `get_date_range`, `truncate_text`, a `Timer` context manager,
`validate_article`, `normalize_score`.

### `main.py`
A leftover stub (`print("Hello from narrative-builder!")`). The real entry point is
`narrative_builder.py`. (`templates/index.html` is an orphaned Flask UI with no backing app ŌĆö
out of scope for the current CLI workflow.)

### `eval/golden_set.jsonl`
Seed reference narratives drawn from the bundled Telangana/Hyderabad corpus. Edit/extend these so
`context_recall`/`context_precision` are meaningful for your data.

### `tests/test_rag.py`
Fast, dependency-light unit tests for `extract_json` (plain / fenced / embedded / garbage) and
`check_grounding` (valid, out-of-range citation, missing citations, fabricated timeline id is
dropped). No vLLM required.

## 3. End-to-end workflows

### A. Generate a narrative (RAG path)
```bash
vllm serve Qwen/Qwen2.5-7B-Instruct --port 8000        # start the model server
uv run python narrative_builder.py --topic "Hyderabad Metro Phase 2B" --top-k 20
```
1. `DataLoader` streams + filters + cleans the dataset.
2. `NewsEmbedder` embeds `full_text`, builds/loads the FAISS index.
3. `RAGNarrator` runs the LangGraph: **retrieve** (FAISS) ŌåÆ **generate** (vLLM, streaming) ŌåÆ
   **validate** (grounding guard, with self-correction retries).
4. `NarrativeEngine` adds heuristic clusters + relationship graph.
5. Final JSON (with `generation_mode: "rag_vllm"`) is printed or written via `--output`.

If vLLM is down or `--no-llm` is passed, steps 3ŌĆō4 use the heuristic engine instead
(`generation_mode: "heuristic"`).

### B. Evaluate hallucination (RAGAS)
```bash
uv run python narrative_builder.py --evaluate --eval-set eval/golden_set.jsonl
# or: uv run python evaluation.py
```
For each topic in the golden set, generate a narrative, collect the retrieved contexts, and score
the answer with RAGAS using a **local judge** (vLLM) and **local embeddings**. Results print as a
table and are saved to `eval_results.json`.

### C. Run unit tests
```bash
uv run pytest tests/
```

## 4. Configuration & setup notes

- **Python is pinned to 3.12** (`.python-version`, `requires-python = ">=3.11,<3.13"`): the
  RAG/eval stack has no 3.14 wheels.
- The **langchain stack is pinned to the 0.3 generation** and **ragas to 0.2.x**: langchain 1.x /
  langchain-community 0.4+ removed a module path that ragas imports.
- vLLM is **not** a Python dependency here ŌĆö the app only talks to its HTTP endpoint via
  `langchain-openai`/`openai`, so the model server can run anywhere reachable at `VLLM_BASE_URL`.
- Auth token: set `VLLM_API_KEY` in `.env`, or export `VLLM_API_KEY`/`OPENAI_API_KEY` to keep the
  secret out of the file. `VLLM_MODEL` must match the server's served-model-name.
- Streaming (`LLM_STREAMING`) affects only the generation call; the graph and the RAGAS judge run
  in normal request/response mode.

## 5. Inner details (deep dive)

This section explains the internal mechanics: data structures, algorithms, control flow, and the
edge cases each component handles.

### 5.1 The canonical article dict
Every downstream component depends on the shape produced by `DataLoader._clean_article`:

```python
{
  "id": <original id or hash(headline)>,   # identity used for citations & timeline matching
  "headline": str,                          # title/headline/text/summary (first non-empty)
  "content": str,                           # story/content/description/text/body
  "full_text": "headline. content",         # what gets embedded AND shown to the LLM
  "date": "YYYY-MM-DDTHH:MM:SS" | "",        # normalised ISO via _parse_date
  "url": str, "source": str, "source_rating": float,
  "author": str, "category": str, "tags": list, "raw": <original record>
}
```
- **Rating extraction** tries `source_rating ŌåÆ sourceRating ŌåÆ rating` (handles schema variants).
- **Source inference**: if `source` is absent, it is derived from the URL domain.
- **Validity gate**: a record is dropped if the headline is missing or shorter than 10 chars.
- The `id` is the join key everywhere: FAISS results, the LLM's `[n]`/`id` citations, the timeline,
  and the graph nodes all resolve back to it.

### 5.2 Ingestion internals (`data_loader.py`)
- `_detect_structure` peeks the first non-whitespace byte: `[` ŌćÆ array (`ijson.items(f,"item")`),
  otherwise it inspects the object for known wrapper keys (`items|articles|data|news`) and parses
  `"<key>.item"`. This streams records **without loading the 81 MB file into memory**.
- `_process_stream` filters by `rating >= min_rating` while streaming; if it yields 0 records it
  falls back to `_load_standard` (full `json.load`) ŌĆö a safety net for unusual structures.
- `_parse_date` tries a list of formats, stripping timezone/fractional parts, and returns the
  original string if nothing matches (never throws).

### 5.3 Embedding & indexing internals (`embedder.py`)
- **Text prep**: `full_text[:EMBEDDING_MAX_LENGTH]` per article, encoded in batches with
  `normalize_embeddings=True`. Normalised vectors make **inner product == cosine similarity**.
- **Caching**: embeddings are pickled to `cache/embeddings.pkl` with the list of article ids. On
  load, the cache is reused only if `len(cached_ids) == len(articles)` (a cheap staleness check).
  *Caveat:* changing `EMBEDDING_MODEL` or the filter without clearing the cache can reuse stale
  vectors ŌĆö delete `cache/embeddings.pkl` when you change either.
- **Index selection** (`_build_faiss_index`): `auto` picks `IndexFlatIP` for <100k articles (exact
  search) or `IndexIVFFlat` otherwise (approximate). For IVF, `nlist` defaults to `sqrt(n)` and it
  must be `train()`-ed before `add()`; `nprobe` controls the speed/recall trade-off. Optional GPU
  via `index_cpu_to_gpu` with CPU fallback on failure.
- **`search`**: encodes the query (normalised), calls `index.search(q, top_k)` ŌåÆ `(similarities,
  indices)`; each in-range index maps to a copied article dict with `similarity_score` attached.

### 5.4 vLLM connection internals (`llm_client.py`)
- **Endpoint/model/key resolution** is role-based. `generator` ŌåÆ `VLLM_*`. `judge` ŌåÆ `RAGAS_JUDGE_*`
  each falling back to the corresponding `VLLM_*` when unset (so one server can do both jobs).
- **Token precedence** (`_resolve_api_key`): explicit non-`EMPTY` config value ŌåÆ `VLLM_API_KEY` env
  ŌåÆ `OPENAI_API_KEY` env ŌåÆ `"EMPTY"`. The OpenAI client requires a non-empty string even when the
  server doesn't check it.
- **`extract_json` algorithm**: (1) try a ```` ```json ```` fenced block; (2) `json.loads` the whole
  candidate; (3) if that fails, scan for the first **balanced** `{...}` by tracking brace depth and
  attempt to parse each candidate block. Returns `None` only if nothing parses ŌĆö callers treat that
  as an empty result (which the grounding guard then flags, triggering a retry).
- **`stream_to_text`**: iterates `llm.stream(messages)`, concatenating `chunk.content`. Because it
  calls `.stream()` directly, streaming is confined to this one call; the client is built **without**
  `streaming=True`, so `.invoke()` elsewhere and the graph's `app.invoke` stay request/response.

### 5.5 LangGraph execution internals (`rag_graph.py`)

**State merging.** `RAGState` is a `TypedDict(total=False)` with no reducers, so LangGraph does a
**shallow last-write-wins merge** of each node's returned dict into the running state. Nodes return
only the keys they change:
- `retrieve` ŌåÆ `{articles, attempts:0}`
- `generate` ŌåÆ `{summary, timeline, attempts: prev+1}`  (the attempts counter lives here)
- `validate` ŌåÆ either `{timeline: cleaned, grounded:True, feedback:""}` (terminal) or
  `{grounded:False, feedback:"- problem\n- problem"}` (retry)

**The prompt** sent in `generate`:
- *System* (`SYSTEM_PROMPT`): the four grounding rules ŌĆö use only the numbered sources, cite `[n]`,
  omit unsupported claims, return a single JSON object.
- *Human*: `Topic: ŌĆ”` + the **numbered context block** + the exact JSON schema to fill + (on a
  retry) `IMPORTANT - fix these problems ŌĆ”` followed by the guard's feedback. The context block
  line format is `[i] (id=ŌĆ”, date=ŌĆ”, source=ŌĆ”)\n<full_text[:RAG_CONTEXT_CHARS]>`, where `i` is the
  1-based citation number and `id` is the article's real id.

**The grounding guard** (`check_grounding`, pure function):
1. `valid_ids = {a["id"]}`, `n = len(articles)`.
2. `cited = {int(m) for m in re.findall(r"\[(\d+)\]", summary)}`.
3. **Out-of-range citation** ŌåÆ problem if any `c < 1 or c > n`.
4. **Missing citations** ŌåÆ problem if there are sources but `cited` is empty.
5. **Fabricated timeline id** ŌåÆ problem for any entry whose `id Ōłē valid_ids`.
6. `clean_timeline` = timeline entries with a valid id (fabricated ones removed).
Returns `{"problems": [...], "clean_timeline": [...]}`.

**The conditional retry** (`_validate_node` + `_should_retry`):
- `grounded = not problems`; `is_final = attempts >= MAX_GENERATION_ATTEMPTS`.
- If `grounded or is_final`: emit the cleaned timeline and route to `END` (on a forced final pass it
  logs the residual problems but still returns clean output ŌĆö the pipeline never emits fabricated
  references).
- Else: store the bulleted `feedback` and route back to `generate`, which appends it to the prompt.
- So the model gets up to `MAX_GENERATION_ATTEMPTS` tries; the guard guarantees the *output* is
  always citation-clean even if the model never fully complies.

### 5.6 Narrative assembly internals (`narrative_engine.py`)
- Articles are sorted by `date` in `__init__`. `_llm_reasons` is a `{id: why_it_matters}` map built
  from `llm_narrative["timeline"]` for O(1) lookup during `build_timeline`.
- **Summary**: returns `llm_narrative["summary"]` if present; otherwise composes a heuristic summary
  from counts, date range (`_get_date_range`), key events, temporal peak (`_analyze_temporal_patterns`),
  top sources, and a narrative-arc sentence (`_determine_narrative_arc`, which compares
  escalation-word density in the first vs last third of the timeline).
- **Timeline**: per article, `why_it_matters = _llm_reasons.get(id) or _generate_importance_reason(article)`
  ŌĆö so LLM reasoning is used where available and the heuristic backfills any gaps.
- **Clusters** (`_cluster_by_keywords`): extract >4-char headline words, take the 20 most common,
  then greedily assign each article to the existing cluster with the largest keyword overlap
  (threshold Ōēź2) or start a new cluster; the theme is the top shared keywords.
- **Graph** (`build_narrative_graph`): a node per article; for each ordered pair it calls
  `_determine_relationship`, which returns `builds_on` (chronological + Ōēź3 shared words),
  `contradicts` (antonym pairs like increase/decrease), `adds_context` (2ŌĆō3 shared words), or
  `escalates` (concernŌåÆcrisis words), each with a weight from `_calculate_edge_weight`.
- These heuristics are **independent of the LLM** and always run, so clusters/graph exist in both
  `rag_vllm` and `heuristic` modes.

### 5.7 Orchestration & fallback internals (`narrative_builder.py`)
`generate_narrative` control flow:
1. `use_llm` true ŌåÆ `try: llm_narrative = _get_narrator().generate(topic, top_k)` and take
   `relevant_articles = llm_narrative["articles"]`.
2. If the summary is empty ŌåÆ discard `llm_narrative` (treat as heuristic).
3. Any exception (e.g. connection refused, timeout) ŌåÆ log a warning, set `llm_narrative=None`,
   `relevant_articles=None` ŌåÆ fall through.
4. If `relevant_articles is None` ŌåÆ `embedder.search(topic, top_k)` (the heuristic retrieval path).
5. Build `NarrativeEngine(relevant_articles, topic, llm_narrative)` and emit the dict with
   `generation_mode = "rag_vllm" if llm_narrative else "heuristic"`.

The narrator is built **lazily** (`_get_narrator`) so the heuristic/`--no-llm` path never imports
langgraph/langchain or contacts vLLM.

### 5.8 RAGAS evaluation internals (`evaluation.py`)
- **Sample schema** (RAGAS 0.2 `EvaluationDataset.from_list`): each row uses
  `user_input` (topic), `response` (generated summary), `retrieved_contexts` (list of truncated
  article texts from `format_contexts`), `reference` (golden answer). Empty contexts are replaced
  with `[""]` so RAGAS doesn't choke.
- **What each metric computes** (conceptually):
  - **faithfulness** ŌĆö the judge LLM decomposes the `response` into atomic claims and checks each
    against `retrieved_contexts`; score = supported claims / total claims. *This is the primary
    hallucination signal* ŌĆö unsupported claims drive it down.
  - **answer_relevancy** ŌĆö the judge generates candidate questions the `response` would answer,
    embeds them, and averages their cosine similarity to `user_input` (penalises off-topic/padded
    answers).
  - **context_precision** ŌĆö judges, per retrieved context, whether it was useful for the
    `reference`, then computes a rank-weighted precision (rewards putting useful contexts first).
  - **context_recall** ŌĆö decomposes the `reference` into statements and checks how many are
    attributable to `retrieved_contexts` (retrieval coverage).
- **Local judge + embeddings**: `LangchainLLMWrapper(get_chat_llm("judge"))` and
  `LangchainEmbeddingsWrapper(HuggingFaceEmbeddings(EMBEDDING_MODEL))` ŌĆö no external API calls.
- **Aggregation**: `result.to_pandas()` ŌåÆ per-row scores; aggregate means via `_safe_mean`
  (drops NaNs, returns `None` if a metric couldn't be computed for any row). Output JSON has
  `{num_samples, aggregate, per_query, results_path}` and is written to `EVAL_RESULTS_PATH`.

### 5.9 Configuration internals (`config.py`)
- **Load order** (later wins): `DEFAULTS` ŌåÆ `.env` (parsed line by line) ŌåÆ `NARRATIVE_<KEY>` OS env
  vars. Unknown keys log a warning and are ignored (typo protection).
- **Type coercion** (`_parse_value`): `true/yes/1/on`ŌåÆ`True`, `false/no/0/off`ŌåÆ`False`,
  `none/null/""`ŌåÆ`None`, then int, then float, else string. So `.env` values land as proper Python
  types (e.g. `LLM_TEMPERATURE=0.1` is a float, `RAGAS_JUDGE_MODEL=null` is `None`).
- `get_config()` caches a singleton; pass a different `--env` to `reload_config()` to swap files.

### 5.10 Failure modes & how they're handled
| Situation | Behaviour |
|---|---|
| vLLM unreachable / wrong port | `generate_narrative` catches it ŌåÆ heuristic narrative (`generation_mode:"heuristic"`) |
| Model returns non-JSON / junk | `extract_json` ŌåÆ `None` ŌåÆ empty summary ŌåÆ guard flags ŌåÆ retry, then heuristic if still empty |
| Model cites `[99]` or a fake id | guard removes/flags it; retry with feedback; final pass strips it |
| `model` name mismatch | vLLM rejects the request ŌåÆ treated as an LLM failure ŌåÆ fallback |
| Stale embedding cache | reused only if id-count matches; delete `cache/embeddings.pkl` after changing model/filter |
| Empty eval reference | row skipped in `load_eval_set`; `context_recall/precision` need references |
